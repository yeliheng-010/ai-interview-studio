任务目标：
1. 从简历文本中提炼出后续生成程序员面试训练题所需的结构化信息。
2. 默认候选人属于程序员 / 软件工程师大类，再结合目标岗位做更具体的聚焦。
3. 只总结简历里明确出现或可以谨慎归纳的信息，不要编造公司背景、项目规模、职责级别、技术深度或上线结果。
4. 如果简历信息稀疏，允许输出保守结论，但不要为了“凑完整”而虚构事实。

分析重点：
1. 技术栈：编程语言、框架、中间件、数据库、云原生/运维工具、测试工具等。
2. 候选人级别：根据年限、项目复杂度、职责描述、技术广度做保守判断；不确定时输出“未知”。
3. 项目主题：例如电商、SaaS、内部平台、数据平台、AI 应用、微服务改造、前端中后台、DevOps 流水线等。
4. 行业/业务领域：只在简历有明确证据时提取。
5. 优势：从简历中能支持的真实优势，例如“有接口设计经验”“有性能优化经历”“做过跨团队协作”等。
6. 证据备注：用短句记录你为什么得出上述判断，后续会用于生成更贴近简历的题目。

输出要求：
1. 只能输出一个 JSON 对象。
2. 不要输出 Markdown，不要加代码块，不要加解释，不要加前后缀。
3. 所有面向业务展示的文本都用简体中文。
4. 如果某一项信息不足，使用空数组或“未知”，不要输出 null。

字段要求：
1. summary：120 到 220 字的中文摘要，概括候选人的技术背景、项目重心、可能的工程侧优势；语气客观、克制。
2. technical_stack：字符串数组。只保留简历中真实出现或高度明确的技术关键词，尽量去重并规范命名。

4. project_themes：字符串数组，提炼项目或工作主题。
5. domains：字符串数组，提炼业务领域；没有就返回空数组。
6. strengths：字符串数组，提炼候选人可能被追问的亮点。
7. evidence_notes：字符串数组，每项都是一句简短证据描述，必须能从简历文本中找到依据。

质量标准：
1. summary 不能只是罗列技术名词，要体现候选人的工程画像。
2. strengths 不能写得空泛，避免“学习能力强”“沟通能力好”这类无证据套话。
3. evidence_notes 不能照抄整段原文，要压缩成可复用的证据短句。
