策略推理原则：
1. 先看简历证据，再决定重点，不要反过来硬套岗位刻板印象。
2. 如果目标岗位是 backend engineer / Java engineer / Python engineer / Go engineer，优先强调接口设计、数据建模、并发、可用性、性能、排障、工程交付。
3. 如果目标岗位是 frontend engineer，优先强调组件设计、状态管理、性能优化、工程化、浏览器与网络、可维护性。
4. 如果目标岗位是 full-stack engineer，重点兼顾前后端协同、数据流、接口边界、交付效率与架构取舍。
5. 如果目标岗位是 DevOps engineer 或 test development engineer，可提高自动化、发布流程、可观测性、稳定性与质量保障相关权重。
6. 面试风格影响提问方式：
   - fundamentals-heavy：更强调基础原理、底层理解、常见概念辨析。
   - project-deep-dive：更强调项目背景、方案选择、复盘与取舍。
   - system-design-heavy：更强调扩展性、稳定性、一致性、容量规划与架构演进。
   - algorithm-heavy：更强调复杂度、数据结构、边界条件与优化思路。
   - real-world-scenario：更强调线上故障、性能瓶颈、协作落地与应急决策。
   - big-tech-style：更强调清晰结构化表达、trade-off、抽象能力、可扩展设计。
   - startup-practical-style：更强调有限资源下的快速交付、成本控制与务实取舍。
   - standard：保持均衡。

1. focus_areas：5 到 8 项，表示题目覆盖的核心能力面，必须可用于后续出题。
2. emphasis：4 到 6 项，描述这套面试要特别强调的判断维度，例如“工程取舍”“故障定位”“性能意识”“边界条件”等。
3. fallback_used：布尔值。如果简历证据不足、不得不回退到通用程序员覆盖，则为 true。
4. difficulty_distribution：固定对象，不要改动。
5. interview_tone：一句简洁中文，描述整套面试的风格和追问方式。

质量标准：
1. focus_areas 不要重复，不要只写“基础”“项目”“设计”这类空词，要写到可直接出题的粒度。
2. emphasis 必须体现面试官真正会观察什么。
3. 如果简历很强，策略要更有针对性；如果简历很弱，策略要更通用但仍具实战训练价值。
4. 不要输出与程序员岗位明显无关的面试方向。
